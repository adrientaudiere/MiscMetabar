Orchestrate an R-package release pipeline. Two modes: **interactive** (default — gated between stages, asks the user before mutating anything) and **`--auto`** (autonomous — runs end-to-end, applies a fixed set of safe deterministic fixes silently, only stops for issues that need human judgment).

## Arguments

`` may contain, in any order:

- A package name (e.g. `MiscMetabar`). If absent, read `Package:` from `DESCRIPTION` in the current working directory.
- `--auto` — autonomous mode. Skips per-stage confirmation for non-destructive stages. Stages 8 and 9 (version bump, merge/push) are still gated.
- `--with-coverage` — include the coverage stage. Off by default (covr is slow).
- `--with-audit` — include the `r-audit` stage. Off by default (slow; produces a TODO file under `audits/`).
- `--with-examples` — also run the slow `rcmdcheck_examples` stage (full `--as-cran` with examples + vignettes). Off by default — the fast `rcmdcheck` stage uses `--no-examples --no-vignettes --no-tests`. Required for a green pre-CRAN verdict.
- `--with-remote-checks` — include `winbuilder` and `rhub`. Off by default (~30 min remote runs; pause-and-resume).
- `--resume` — read the existing state file and pick up at `current_stage`. Useful after a `--with-remote-checks` pause or a Ctrl-C.
- `--no-cache` — force every stage to re-run, ignoring source-hash cache hits.

When invoked from the pqverse workspace and only a package name is given, resolve `pqverse_pkg/{name}/`.

Path: `<workspace>/audits/{pkg}_release-state.json`. Schema: `~/.claude/skills/r-pkg-release/state-schema.json`. Helpers: `~/.claude/skills/r-pkg-release/state.sh`.

`status ∈ {PENDING, RUNNING, DONE, FAILED, SKIPPED, AWAITING_REMOTE}`.

## Stage cache (CRITICAL — read this first)

Long stages (`tests`, `coverage`, `audit`, `rcmdcheck`, `rcmdcheck_examples`) run for minutes-to-tens-of-minutes against real pqverse packages. Re-running them on every invocation defeats the purpose of `--auto`.

**Cache layout:** `<workspace>/audits/.release-cache/{pkg}/{stage}/`

Per stage:
- `<ts>.log` — full stdout+stderr capture (the agent **must** Read this file rather than re-running the command).
- `<ts>.rds` — R result object (rcmdcheck or testthat).
- `<ts>.classified.json` (rcmdcheck) or `<ts>.summary.json` (tests) — parser output.
- `<ts>.signature` — sha256 over the relevant source files for that stage.
- `latest` — text file holding the newest `<ts>`.

**Cache hit policy** (source-hash based):

```bash
. ~/.claude/skills/r-pkg-release/cache.sh
if [[ "$NO_CACHE" != "true" ]] && cache_is_fresh "$PKG" "$PKG_DIR" "$STAGE"; then
  ts="$(cache_latest "$PKG" "$STAGE")"
  log="$(cache_path_for "$PKG" "$STAGE" "$ts" log)"
  json="$(cache_path_for "$PKG" "$STAGE" "$ts" classified.json)"  # or summary.json
  # Mark stage DONE from cache; the agent reads $log / $json directly.
fi
```

`cache_is_fresh` is true iff:
1. A `latest` pointer exists for this stage, AND
2. The cached `.signature` matches the current source signature, AND
3. The corresponding `.log` file is on disk.

Source signatures use different file globs per stage (see `cache.sh:_cache_globs_for_stage`):
- `format`: `R/**/*.R`
- `document`: `R/**/*.R + DESCRIPTION + NAMESPACE`
- `rcmdcheck`: `R/ + man/ + tests/ + DESCRIPTION + NAMESPACE`
- `rcmdcheck_examples`, `winbuilder`, `rhub`: above + `vignettes/ + inst/`
- `tests`, `coverage`: `R/ + tests/ + DESCRIPTION + NAMESPACE`
- `audit`: `R/ + DESCRIPTION + NAMESPACE`
- `cran_extra`: `DESCRIPTION + NEWS.md + README.* + cran-comments.md`

If the user passes `--no-cache`, every stage re-runs.

After a stage runs (cache miss), record the cache:

```bash
ts="$(cache_new "$PKG" "$STAGE")"
sig="$(cache_signature "$PKG_DIR" "$STAGE")"
# ... pass --cache-dir "$(cache_dir $PKG $STAGE)" --ts "$ts" to the R script
cache_record "$PKG" "$STAGE" "$ts" "$sig"
state_set_cache "$STATE" "$STAGE" "$(cache_path_for $PKG $STAGE $ts log)" "$sig" false
```

## Output-handling rule (READ FILES, DON'T RE-PIPE)

Every long-running R command **must** redirect output to the cache `.log` file (the `--cache-dir` + `--ts` mechanism does this automatically). To inspect the result, the agent **must use the Read tool** on the `.log`/`.json`/`.rds` files. **Do not** re-invoke the command via Bash to grep its output — that re-runs the long stage.

Right pattern:
```
1. Run: Rscript parse-rcmdcheck.R run <pkg> --cache-dir <dir> --ts <ts>
2. Read tool: <dir>/<ts>.classified.json    ← inspect parser verdict
3. Read tool: <dir>/<ts>.log                ← look up specific failures
```

Wrong pattern (forbidden):
```
Bash: Rscript -e 'rcmdcheck::rcmdcheck(...)' | grep ERROR    # re-runs everything
Bash: Rscript -e 'devtools::test(...)' 2>&1 | tail -50        # ditto
```

## Stages and order

`format` → `document` → `rcmdcheck` → `tests` → `coverage` (opt) → `audit` (opt) → `rcmdcheck_examples` (opt) → `cran_extra` → `winbuilder` (opt) → `rhub` (opt) → `bump_version` (gated) → `merge_push` (gated).

Stages marked `SKIPPED` at init time stay skipped (no flag → no run).

## `--auto` per-stage loop

For each non-skipped stage in order:

1. Read state. If `DONE` or `SKIPPED`, advance. If `AWAITING_REMOTE`, prompt user for the remote result, parse, then advance.
2. **Cache check.** If `--no-cache` is not set: `cache_is_fresh "$PKG" "$PKG_DIR" "$STAGE"`. If fresh, mark stage `DONE` (with `from_cache=true`), Read the cached `.classified.json` / `.summary.json` to surface counts to the user, advance.
3. `state.sh set-stage <stage> RUNNING`.
4. Mint a cache stamp and run the stage with `--cache-dir <dir> --ts <ts>`. The R script writes `.log`, `.rds`, and `.classified.json` / `.summary.json` to the cache.
5. **Use the Read tool** on the cache files to inspect the result. Never re-invoke the R command to grep its output.
6. Parse the JSON → list of issues, each classified `AUTO_FIX:<fixer>` or `AMBIGUOUS`.
7. For each `AUTO_FIX`, run `Rscript ~/.claude/skills/r-pkg-release/auto-fixers.R <fixer> <pkg_dir>`. Record via `state.sh add-fix <stage> <fixer>`.
8. If at least one fix was applied this iteration, **retry the stage once** — note the retry invalidates the previous cache stamp because source has changed; `cache_signature` will differ.
9. If `AMBIGUOUS` is non-empty:
   - For each: `state.sh mark-blocking <stage> <severity> <type> <message> [location]`.
   - `state.sh set-stage <stage> FAILED`.
   - **Stop the run.** Report the blocking issues, citing the `.log` line range so the user can jump there. The PreToolUse hook will now block any subsequent `git commit/push/merge` until the state is cleared.
10. Otherwise `cache_record` + `state.sh set-cache` + `state.sh set-stage <stage> DONE` and advance.

Stages 8 and 9 are **always interactive**, even in `--auto` — destructive actions confirm.

## Stage helpers

All long stages share the same cache-aware shape:

```bash
. ~/.claude/skills/r-pkg-release/cache.sh
. ~/.claude/skills/r-pkg-release/state.sh

if [[ "$NO_CACHE" != "true" ]] && cache_is_fresh "$PKG" "$PKG_DIR" "$STAGE"; then
  ts="$(cache_latest "$PKG" "$STAGE")"
  log="$(cache_path_for "$PKG" "$STAGE" "$ts" log)"
  state_set_cache "$STATE" "$STAGE" "$log" "$(cache_signature $PKG_DIR $STAGE)" true
  state_set_stage "$STATE" "$STAGE" DONE "from cache"
  # Read tool: the .classified.json / .summary.json to surface counts.
else
  ts="$(cache_new "$PKG" "$STAGE")"
  dir="$(cache_dir "$PKG" "$STAGE")"
  state_set_stage "$STATE" "$STAGE" RUNNING
  Rscript <stage-script> --cache-dir "$dir" --ts "$ts" "$PKG_DIR"
  sig="$(cache_signature "$PKG_DIR" "$STAGE")"
  cache_record "$PKG" "$STAGE" "$ts" "$sig"
  state_set_cache "$STATE" "$STAGE" "$dir/$ts.log" "$sig" false
  # Read tool: $dir/$ts.classified.json (or .summary.json) to classify.
fi
```

### `format` (fast)

```bash
Rscript ~/.claude/skills/r-pkg-release/auto-fixers.R fix_format "$PKG_DIR" \
  > "$dir/$ts.log" 2>&1
```

Idempotent. Cache hit policy: source-hash on `R/*.R`. No issues generated — mark `DONE`.

### `document` (fast)

```bash
Rscript ~/.claude/skills/r-pkg-release/auto-fixers.R fix_document "$PKG_DIR" \
  > "$dir/$ts.log" 2>&1
```

Idempotent. On error, the `.log` contains the message; classify as `AMBIGUOUS` (likely an `@param` mismatch).

### `rcmdcheck` (fast pass — no examples, no vignettes, no tests)

```bash
Rscript ~/.claude/skills/r-pkg-release/parse-rcmdcheck.R run "$PKG_DIR" \
  --cache-dir "$dir" --ts "$ts"
```

Writes `<dir>/<ts>.log` (raw rcmdcheck output), `.rds` (result object), `.classified.json` (parser verdict). The agent **Reads** the `.classified.json` to get `{summary, auto_fix, ambiguous}`, and `.log` only when it needs to look up a specific failure line.

Classifier rules (see `parse-rcmdcheck.R`):

| Pattern | Severity | Action |
|---|---|---|
| `non-ASCII` | NOTE | `fix_non_ascii` |
| `Encoding not declared` | NOTE | `fix_encoding_declaration` |
| `code/documentation mismatches`, stale `.Rd` | WARNING | `fix_document` |
| `Undocumented arguments`, missing `@return` | WARNING | AMBIGUOUS |
| `examples > 5 seconds`, examples failed | NOTE/WARNING | AMBIGUOUS |
| any `ERROR` | ERROR | AMBIGUOUS |
| anything else | * | AMBIGUOUS (default-deny) |

### `tests` (potentially long)

```bash
Rscript ~/.claude/skills/r-pkg-release/parse-tests.R run "$PKG_DIR" \
  --cache-dir "$dir" --ts "$ts"
```

Writes `<dir>/<ts>.log`, `.rds` (testthat result), `.summary.json` (per-file pass/fail + a `failing` list). The agent Reads `.summary.json` first; if anything failed, it Reads the relevant range of `.log` to surface the test message.

After a partial fix (user resolved one failing test), **re-run only the failing files** with `--filter "<regex over failing files>"` rather than the whole suite. The result is also cached.

Failing tests are always `AMBIGUOUS`.

### `coverage` (only if `--with-coverage` — long)

```bash
Rscript -e 'sink(file("'$dir'/'$ts'.log", "wt"), split = FALSE);
```

### `rcmdcheck_examples` (only if `--with-examples` — long, full --as-cran)

Same shape as `rcmdcheck` but runs the full `--as-cran` (examples + vignettes). Source signature is broader (includes `vignettes/` and `inst/`) so this stage's cache is independent of the fast pass.

Load and follow `r-lib:cran-extrachecks`. Capture its output to `<dir>/<ts>.log`. Issues that the skill itself flags as blocking → `AMBIGUOUS`. Suggestions → ignore in `--auto`.

## Resume

`--resume` reads `current_stage` and re-enters the loop there. Stages already `DONE` or `SKIPPED` are not re-run. If the previous run hit `FAILED` with `AMBIGUOUS`, the user must clear `has_blocking_errors` (the hook tells them how) and remove the entries from `ambiguous_issues` before the resume can advance past that stage.

## Important rules

- **Read cached files; never re-pipe bash.** Every long stage writes its full output to `audits/.release-cache/{pkg}/{stage}/<ts>.log` plus a structured `.json`/`.rds`. Use the Read tool on those files. Re-invoking `Rscript -e '...' | grep ERROR` defeats the cache and re-runs minutes-long commands.
- **Trust the source-hash cache.** If `cache_is_fresh` is true, the stage's verdict from a previous run is still valid; mark `DONE (cached)` and move on. Only bypass with `--no-cache`.
- **Default-deny in the classifier:** anything we don't recognize is `AMBIGUOUS`. Never silently apply a fix the agent doesn't understand.
- **Stages 8 and 9 always confirm with the user**, even in `--auto`. The `--auto` flag means "no confirmation for non-destructive stages", not "no confirmation ever".
- The PreToolUse hook (`hooks/pretooluse-block-on-error.sh`) is project-installed once via `hooks/settings-snippet.json`; it stays silent unless `has_blocking_errors == true` in the most recent state file under `audits/`.
- Do not retry pushes or merges with `--force`. Surface the error and stop.
- Keep stage summaries concise — the cached files have the detail.

# CRAN Extra Checks

Help R package developers prepare packages for CRAN submission by systematically checking for common ad-hoc requirements that CRAN reviewers enforce but `devtools::check()` doesn't catch.

## Workflow

1. **Initial Assessment**: Ask user if this is first submission or resubmission
2. **Run Standard Checklist**: Work through each item systematically (see below)
3. **Identify Issues**: As you review files, note specific problems
4. **Propose Fixes**: Suggest specific changes for each issue found
5. **Implement Changes**: Make edits only when user approves
6. **Verify**: Confirm all changes are complete

## Standard CRAN Preparation Checklist

Work through these items systematically:

1. **Create NEWS.md**: Run `usethis::use_news_md()` if not already present
2. **Create cran-comments.md**: Run `usethis::use_cran_comments()` if not already present
3. **Review README**:
   - Ensure it includes install instructions that will be valid when the package is accepted to CRAN (usually `install.packages("pkgname")`).
   - Check that it does not contain relative links. This works on GitHub but will be flagged by CRAN. Use full URLs to package documentation or remove the links.
   - Does the README clearly explain the package purpose and functionality?
   - **Important**: If README.Rmd exists, edit ONLY README.Rmd (README.md will be overwritten), then run `devtools::build_readme()` to re-render README.md
4. **Proofread DESCRIPTION**: Carefully review `Title:` and `Description:` fields (see detailed guidance below)
5. **Check function documentation**: Verify all exported functions have `@return` and `@examples` (see detailed guidance below)
6. **Verify copyright holder**: Check that `Authors@R:` includes a copyright holder with role `[cph]`
7. **Review bundled file licensing**: Check licensing of any included third-party files
8. **Run URL checks**: Use `urlchecker::url_check()` and fix any issues

## Detailed CRAN Checks

### Documentation Requirements

**Return Value Documentation (Strictly Enforced)**

CRAN now strictly requires `@return` documentation for all exported functions. Use the roxygen2 tag `@return` to document what the function returns.

- Required even for functions marked `@keywords internal`
- Required even if function returns nothing - document as `@return None` or similar
- Must be present for every exported function

Example:
```r
# Missing @return - WILL BE REJECTED
#' Calculate sum
#' @export
my_sum <- function(x, y) {
  x + y
}

# Correct - includes @return
#' Calculate sum
#' @param x First number
#' @param y Second number
#' @return A numeric value
#' @export
my_sum <- function(x, y) {
  x + y
}

# For functions with no return value
#' Print message
#' @param msg Message to print
#' @return None, called for side effects
#' @export
print_msg <- function(msg) {
  cat(msg, "\n")
}
```

**Examples for Exported Functions**

If your exported function has a meaningful return value, it will almost definitely require an `@examples` section. Use the roxygen2 tag `@examples`.

- Required even for functions marked `@keywords internal`
- Exceptions exist for functions used purely for side effects (e.g., creating directories)
- Examples must be executable

**Un-exported Functions with Examples**

If you write roxygen examples for un-exported functions, you must either:

1. Call them with `:::` notation: `pkg:::my_fun()`
2. Use `@noRd` tag to suppress `.Rd` file creation

**Using `\dontrun{}` Sparingly**

`\dontrun{}` should only be used if the example really cannot be executed (e.g., missing additional software, API keys, etc.).

- If showing an error, wrap the call in `try()` instead
- Consider custom predicates (e.g., `googlesheets4::sheets_has_token()`) with `if ()` blocks
- Sometimes `interactive()` can be used as the condition
- Lengthy examples (> 5 sec) can use `\donttest{}`

**Never Comment Out Code in Examples**

```r
# BAD - Will be rejected
#' @examples
#' # my_function(x)  # Don't do this!
```

CRAN's guidance: "Examples/code lines in examples should never be commented out. Ideally find toy examples that can be regularly executed and checked."

**Guarding Examples with Suggested Packages**

Use `@examplesIf` for entire example sections requiring suggested packages:

```r
#' @examplesIf rlang::is_installed("dplyr")
#' library(dplyr)
#' my_data %>% my_function()
```

For individual code blocks within examples:
```r
#' @examples
#' if (rlang::is_installed("dplyr")) {
#'   library(dplyr)
#'   my_data %>% my_function()
#' }
```

### DESCRIPTION Title Field

CRAN enforces strict Title requirements:

**Use Title Case**

Capitalize all words except articles like 'a', 'the'. Use `tools::toTitleCase()` to help format.

**Avoid Redundancy**

Common phrases that get flagged:
- "A Toolkit for" → Remove
- "Tools for" → Remove
- "for R" → Remove

Examples:
```r
# BAD
Title: A Toolkit for the Construction of Modeling Packages for R

# GOOD
Title: Construct Modeling Packages

# BAD
Title: Command Argument Parsing for R

# GOOD
Title: Command Argument Parsing
```

**Quote Software/Package Names**

Put all software and R package names in single quotes:

```r
# GOOD
Title: Interface to 'Tiingo' Stock Price API
```

**Length Limit**

Keep titles under 65 characters.

### DESCRIPTION Description Field

**Never Start With Forbidden Phrases**

CRAN will reject descriptions starting with:
- "This package"
- Package name
- "Functions for"

```r
# BAD
Description: This package provides functions for rendering slides.
Description: Functions for rendering slides to different formats.

# GOOD
Description: Render slides to different formats including HTML and PDF.
```

**Expand to 3-4 Sentences**

Single-sentence descriptions are insufficient. Provide a broader description of:
- What the package does
- Why it may be useful
- Types of problems it helps solve

```r
# BAD (too short)
Description: Render slides to different formats.

# GOOD
Description: Render slides to different formats including HTML and PDF.
    Supports custom themes and progressive disclosure patterns. Integrates
    with 'reveal.js' for interactive presentations. Designed for technical
    presentations and teaching materials.
```

**Quote Software Names, Not Functions**

```r
# BAD
Description: Uses 'case_when()' to process data.

# GOOD
Description: Uses case_when() to process data with 'dplyr'.
```

Software, package, and API names get single quotes (including 'R'). Function names do not.

**Expand All Acronyms**

All acronyms must be fully expanded on first mention:

```r
# BAD
Description: Implements X-SAMPA processing.

# GOOD
Description: Implements Extended Speech Assessment Methods Phonetic
    Alphabet (X-SAMPA) processing.
```

**Publication Titles Only in Double Quotes**

Only use double quotes for publication titles, not for phrases or emphasis:

```r
# BAD
Description: Handles dates like "the first Monday of December".

# GOOD
Description: Handles dates like the first Monday of December.
```

### URL and Link Validation

**All URLs Must Use HTTPS**

CRAN requires `https://` protocol for all URLs. HTTP links will be rejected.

```r
# BAD
URL: http://paleobiodb.org/

# GOOD
URL: https://paleobiodb.org/
```

**No Redirecting URLs**

CRAN rejects URLs that redirect to other locations. Example rejection:

```
Found the following (possibly) invalid URLs:
URL: https://h3geo.org/docs/core-library/coordsystems#faceijk-coordinates
     (moved to https://h3geo.org/docs/core-library/coordsystems/)
```

**Use urlchecker Package**

```r
# Find redirecting URLs
urlchecker::url_check()

# Automatically update to final destinations
urlchecker::url_update()
```

Bump an R package from `X.Y.Z.9xxx` (dev) to a release version `X.Y.Z`. This is a thin wrapper around `usethis::use_version()`. It mutates two files (`DESCRIPTION`, `NEWS.md`) and **does not commit**.

## Arguments

`MiscMetabar patch --auto` may contain, in any order:

- A package name (e.g. `MiscMetabar`). If absent, read `Package:` from `DESCRIPTION` in the current working directory.
- A bump type: `patch` | `minor` | `major`. If absent and `--auto` is set, default to `patch`. If absent and not `--auto`, ask the user once.
- `--auto` — non-interactive mode (no `AskUserQuestion`).

When invoked from the pqverse workspace and only a package name is given, resolve the package directory to `pqverse_pkg/{name}/`.

## Pre-flight (stop on first failure)

1. `DESCRIPTION` exists; extract `Package` and `Version`.
2. Current version ends in `.9xxx`. If not, stop and report — the package is already at a release version.
3. Current branch is `dev`. If not, stop and report.
4. Working tree is clean for `DESCRIPTION` and `NEWS.md` (`git diff --quiet DESCRIPTION NEWS.md`). If dirty, stop and report.
5. `usethis` is installed:
   ```bash
   Rscript -e 'invisible(packageVersion("usethis"))'
   ```
   If missing, stop and tell the user to `install.packages("usethis")`. **No fallback** — `usethis::use_version()` is the single supported method.

## Compute the new version

Strip the `.9xxx` suffix, then increment the chosen component, zeroing lower components:

| Current | Type | New |
|---|---|---|
| `1.2.3.9000` | major | `2.0.0` |
| `1.2.3.9000` | minor | `1.3.0` |
| `1.2.3.9000` | patch | `1.2.4` |
| `0.16.0.9001` | minor | `0.17.0` |

Display: `Bumping {pkg}: {current} → {new}`.

## Apply the bump

```bash
Rscript -e "usethis::use_version('${NEW_VERSION}')"
```

`usethis::use_version()` is non-interactive when called from `Rscript`: it updates `DESCRIPTION$Version`, rewrites the first `NEWS.md` heading to `# {pkg} {new_version}`, and skips its commit prompt (leaves changes staged-but-uncommitted).

If `use_version()` *did* auto-commit on this system, immediately undo with `git reset --soft HEAD~1` so Stage 9 can produce its own `Release {pkg} {version}` commit.

## Show the diff and stop

```bash
git --no-pager diff DESCRIPTION NEWS.md
```

Report:

> "DESCRIPTION + NEWS.md updated to {new}. **No commit made.** Run `/r-pkg-merge-release {pkg}` (Stage 9) to commit, merge dev → master, push, and bump dev for the next cycle. To revert: `git restore DESCRIPTION NEWS.md`."

## Important rules

- Do not commit, push, or change branches.
- Do not invoke Stage 9.
- No heuristics on the bump type — caller decides (or accepts the `patch` default in `--auto`).
- No fallback chain — if `usethis` isn't installed, stop and ask the user to install it.
- The first `NEWS.md` heading is rewritten to `# {pkg} {new}`. Any custom suffix or date on that line is dropped. The `[CRAN]` tag is re-applied later by Stage 9's `merge-release.sh` based on a live CRAN check.
